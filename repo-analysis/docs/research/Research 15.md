# Further research into sufficiency estimatation, artifact-level attribution, repository-native memory, execution-derived signals, evaluation protocols

**the most promising next-generation repo agent is not a better retriever alone.**
It is an **evidence-control system** with five tightly connected parts:

1. a **sufficiency estimator** that knows when to stop searching,
2. **artifact-level attribution** that knows what actually mattered,
3. **repository-native memory** that persists project state over time,
4. **execution-derived intermediate signals** that feed back into retrieval and planning, and
5. **evaluation protocols** that make shortcutting and benchmark inflation harder. Recent academic work increasingly supports this as the right framing. ([arXiv][1])

On **sufficiency estimation**, the strongest new signal is that the field is moving from “retrieve relevant context” to “retrieve the **smallest sufficient** context.” The 2026 compression paper on issue resolution proposes **OCD**, which explicitly identifies the *smallest context subsequence* that still enables a correct patch, and reports that its compression method remains stable while improving resolution rates over earlier compression baselines on SWE-bench Verified. Repoformer had already shown that retrieval should be **selective rather than automatic**, and RepoShapley adds that chunk utility is often **interaction-dependent**, so deciding whether to keep or drop context should consider coalition effects, not just one chunk at a time. The strongest open problem now is not simple ranking; it is learning a **stop policy** that predicts when the expected gain from another retrieval step is lower than its token and latency cost. ([arXiv][1])

On **repository-native memory**, the evidence is much stronger than before. CodeMEM shows that code-agent memory works better when it is **AST-guided and code-centric** rather than natural-language-centric, improving instruction following on both the current turn and session level while reducing interaction rounds. Another line of work builds memory directly from the repository’s own history: *Improving Code Localization with Repository Memory* uses recent commits, linked issues, and functionality summaries of active parts of the codebase to improve localization on SWE-bench Verified and SWE-bench Live. MemGovern then shows that curated “experience cards” distilled from GitHub history improve SWE-bench Verified resolution, and subtask-level memory improves Pass@1 by aligning retrieved memory to the current subproblem instead of whole-instance similarity. The practical implication is that the right memory object is probably **not conversation history**, but a structured store of design commitments, active-module summaries, historical fix patterns, regression history, and evolving spec state. ([arXiv][2])

That point gets even stronger in long-horizon settings. SWE-ContextBench shows that correctly selected summarized prior experience can improve resolution accuracy while reducing runtime and token cost, while bad or unfiltered reuse can hurt. SLUMP shows a more specific failure mode: when the specification emerges gradually instead of being given up front, faithfulness drops, and its ProjectGuard layer recovers **90% of the faithfulness gap** on Claude Code, increases fully faithful components from **118 to 181**, and reduces severe failures from **72 to 49**. That is unusually strong evidence that **specification tracking** should be treated as a first-class memory problem. ([arXiv][3])

On **utilization-aware attribution**, the field has made diagnostic progress but not yet solved the control problem. ContextBench made the retrieved-versus-utilized gap explicit, and the newer lines of work are starting to quantify *which* artifacts were actually helpful. RepoShapley is the clearest example: it supervises keep/drop decisions using **Shapley-style marginal contributions**, explicitly modeling that some snippets only help in combination and some hurt decoding. RACE-bench complements this by adding intermediate reasoning supervision for repository-level feature addition; it reports that “apply-success but test-fail” cases have **35.7% lower reasoning recall** and **94.1% higher over-prediction** than successful ones. That means the field can increasingly detect when agents gathered context but failed to convert it into grounded implementation steps. What still seems missing is a clean **artifact-to-outcome support graph** that can say which retrieved files, symbols, traces, and memories actually changed the probability of the final correct patch. ([arXiv][4])

On **execution evidence as reusable signals**, the strongest shift is from using tests only as final pass/fail judges to using runtime evidence as a reusable intermediate representation. DAIRA is the clearest current example: it integrates dynamic analysis into the issue-resolution loop, extracts runtime evidence like **variable mutations and call stacks**, converts them into structured semantic reports, and reports **79.4%** on SWE-bench Verified while reducing inference expense by about **10%** and input token consumption by about **25%**. EnvGraph applies a similar principle to repository-level generation by treating repository executability as **environment alignment** and using **execution-evidence-based attribution** to guide iterative revision, improving functional correctness by **5.72–5.87 points** over the strongest non-EnvGraph baseline. These papers suggest that execution signals should be compressed into reusable objects such as caller paths, violated assumptions, environment mismatches, state transitions, and regression signatures, then fed back into retrieval, planning, and memory. ([arXiv][5])

Feature-level evaluation explains why this matters. FeatureBench derives tasks from unit tests and dependency graphs, builds **200 tasks** and **3,825 executable environments** from **24 repositories**, and shows that Claude 4.5 Opus drops from **74.4%** on SWE-bench to **11.0%** on FeatureBench. That gap strongly suggests that static repository understanding and true **feature-level executable understanding** are different capabilities. Execution-derived signals are likely one of the best ways to close that gap without simply dumping more code into context. ([arXiv][6])

On **evaluation**, the evidence now strongly supports your instinct that this is foundational, not bookkeeping. SWE-QA-Pro and IDE-Bench both push toward more trustworthy settings: SWE-QA-Pro uses **long-tail repositories with executable environments** and filters out questions solvable without real exploration, while IDE-Bench uses **80 tasks across eight never-published repositories** to reduce contamination and emulate IDE-native agent workflows. At the same time, *The SWE-Bench Illusion* shows that state-of-the-art models can identify buggy file paths from issue descriptions alone at up to **76%** on SWE-bench Verified but only **53%** on repositories outside SWE-bench, which is strong evidence that memorization or benchmark-specific artifacts can inflate apparent ability. ([arXiv][7])

Even executable evaluation is not enough if the tests are weak. STING finds that **77%** of SWE-bench Verified instances contain at least one surviving incorrect variant under the original suites, adds **1,014** validated tests across **211** instances, and lowers the resolved rates of the top-10 repair agents by **4.2%–9.0%** after strengthening the suites. SWE-ABS pushes the same direction with adversarial strengthening and reports that about **one in five** previously “solved” patches are semantically incorrect under stronger testing. So the field still needs benchmarks that are simultaneously **realistic, executable, contamination-resistant, and test-strong**. ([arXiv][8])

My updated ranking still stays the same, but now with sharper reasons:

**1. Sufficiency estimation + stopping** because it is the most direct lever on token cost and latency, and the literature has good ingredients but no complete solution yet. ([arXiv][1])
**2. Repository-native memory** because long-horizon engineering clearly breaks without a durable project-state layer. ([arXiv][3])
**3. Utilization-aware attribution** because retrieval gains are hard to control if you cannot tell what evidence actually influenced the result. ([arXiv][4])
**4. Execution evidence reuse** because it looks like one of the best ways to raise behavioral correctness without brute-force context expansion. ([arXiv][5])
**5. Leakage-resistant, process-aware evaluation** because otherwise improvements in the first four areas are too easy to misread. ([arXiv][7])

So the strongest conclusion after this deeper pass is:

**the next big step for repository agents is not “more context.”**
It is building a system that can **stop earlier, remember better, explain what mattered, learn from execution, and be measured under harder and cleaner protocols.** ([arXiv][1])
