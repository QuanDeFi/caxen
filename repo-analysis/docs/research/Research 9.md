# Further research into Evidence retrieval, Evidence utilization metrics, Project-state memory, feature reasoning

**the original problem is no longer best framed as “how do we retrieve better repo context?”**
It is now better framed as **how to build a coding agent that can identify the smallest sufficient evidence set, prove that it actually used that evidence, preserve project state across long evolving sessions, ground its reasoning in executable feature behavior, and be evaluated under protocols that do not reward memorization or prompt leakage.** The newer papers make those five layers look like distinct research problems, not just implementation details. ([arXiv][1])

## 1. Minimal-sufficient evidence retrieval

The deeper research strongly supports a move away from “maximize recall” and toward **minimal sufficient evidence retrieval**. ContextBench shows that current agents still lean toward **recall over precision** and leave a substantial gap between **explored** and **utilized** context. The more general “Sufficient Context” paper sharpens the conceptual split: some failures happen because the retrieved context was not enough, while others happen even when the context was sufficient but poorly used. That makes “context sufficiency” itself a measurable target rather than a vague intuition. ([arXiv][1])

Several newer methods support the same direction from different angles. SWE-Pruner shows that task-aware pruning can cut agent-context tokens by **23–54%** on SWE-style tasks and compress single-turn code contexts by up to **14.84×** with minimal performance loss. Hierarchical Context Pruning shows that preserving dependency topology while removing low-value implementation detail can shrink repository prompts substantially without hurting, and often improving, completion quality. Repoformer argues that retrieval itself should be **conditional**, not automatic on every query, and reports large inference-speed gains from selective retrieval. FastCode reaches a similar conclusion at the agent level by separating **structural scouting** from **full-text consumption**, beating baselines while using fewer tokens. ([arXiv][2])

The most interesting new evidence is that **not all retrieved context is merely neutral noise; some of it is actively harmful**. Impact-driven Context Filtering introduces a chunk-level likelihood metric and finds that in top-10 retrieved cross-file contexts, only about **15%** genuinely help completion, while **5.6%** are harmful and **85%** overall are non-contributory or worse. That is one of the clearest signals that retrieval research should move from relevance ranking toward **benefit prediction**. In other words, the open problem is not just “what looks relevant?” but “what will actually increase the probability of a correct edit or answer?” ([arXiv][3])

This also changes what a strong repo retriever should look like. The best recent systems are increasingly **dependency-aware and stopping-aware**. CoSIL uses iterative graph search over module and function call graphs and improves localization enough to boost downstream issue resolution. GraphLocator adds causal issue graphs to handle symptom-to-cause and one-to-many mismatch cases, improving localization precision and recall and downstream resolving performance. InlineCoder adds a different twist: it generates an initial “anchor,” uses it to inline upstream callers and downstream callees, and then uses **confidence estimation** to assemble a better prompt. These systems point toward a retrieval stack that behaves more like **hypothesis testing over repository structure** than flat RAG. ([arXiv][4])

So the real open questions here are now quite sharp: how to estimate **when enough evidence has been found**, how to detect **harmful context before paying to read it**, how to route search through **dependency and causal structure**, and how to design stop policies that minimize tokens without cutting off essential evidence. That is a more specific and stronger version of the original repo-understanding problem. ([arXiv][1])

## 2. Evidence utilization metrics and mechanisms

This direction now looks like a research area in its own right. ContextBench is important because it makes the **retrieved-versus-used** gap explicit. It logs the regions agents inspect and compares them to gold context at file, block, and line level, which lets it diagnose behavior beyond final issue resolution. The big implication is that retrieval metrics alone are not enough; a system can fetch relevant code and still fail because it never truly converts that evidence into grounded reasoning or edits. ([arXiv][1])

ReCUBE pushes this much further by isolating **repository-level context utilization** directly. Instead of asking an agent to solve a broad issue, it masks a file inside a real repository and asks the model to reconstruct it using the rest of the repository, dependency specs, and docs. The benchmark uses **usage-aware tests** that check both internal logic and cross-file integration. Even there, GPT-5 reaches only **37.57% strict pass rate** in the full-context setting, and the paper’s caller-centric exploration toolkit improves strict pass rate by up to **7.56%**. That is strong evidence that the current bottleneck is not only retrieval quality but the model’s ability to **use structural evidence correctly once retrieved**. ([arXiv][5])

The most useful methodological addition here is the move toward **chunk-level and path-level attribution**. Impact-driven Context Filtering proposes a likelihood-based metric for the effect of each retrieved chunk. ReCUBE contributes a usage-aware executable benchmark for whether retrieved repo context actually translates into correct behavior. InlineCoder adds a practical inference mechanism: generate an anchor first, then use caller/callee inlining plus perplexity-based confidence to assemble a context that is easier for the model to exploit. Together, these suggest that the next wave of work should focus on **evidence attribution**, **utilization-aware reranking**, and **confidence-triggered abstention or continued search**. ([arXiv][3])

So the open problem is now narrower and more actionable than before: build systems that can say **which retrieved artifacts caused the answer or patch**, detect when the answer is still under-supported, and turn repository structure into prompts that the model can actually exploit rather than merely observe. That is a direct expansion of your original question, because “higher accuracy with fewer tokens” depends as much on **evidence use** as on evidence retrieval. ([arXiv][1])

## 3. Project-state memory for evolving specifications

This is the area that most clearly expands the original framing. SWE-ContextBench shows that coding tasks should not be treated as independent if you want to understand how agents work over time. It augments base tasks with related tasks and finds that **correctly selected summarized experience** improves accuracy while reducing runtime and token cost, especially on harder tasks, whereas unfiltered or wrongly selected experience can hurt. That means memory is not just “more context later”; it is a separate problem of **what to remember, how to summarize it, and when to reuse it**. ([arXiv][6])

SLUMP goes beyond reuse and exposes a deeper failure mode: when the full spec is not available upfront and instead emerges gradually, structural integration degrades. Its ProjectGuard mitigation is one of the strongest pieces of evidence in this whole area: an external project-state layer recovers **90%** of the faithfulness gap on Claude Code, increases fully faithful components from **118 to 181**, and reduces severe failures from **72 to 49**. That suggests that long-horizon repo reasoning is not only a retrieval problem but a **state-consistency** problem. The agent needs a persistent representation of commitments, invariants, and repository organization that survives across turns and restarts. ([arXiv][7])

Long-horizon evaluation papers reinforce that point. LoCoEval contains repository-oriented conversations averaging **50 turns** and up to **64K–256K tokens**, and its main result is sobering: simple Vanilla RAG often beats more elaborate memory systems, while general-purpose memory systems underperform and vary unstably across backbones. The paper’s conclusion is not that memory is useless, but that current memory designs are badly adapted to repository-oriented conversations and need tighter integration with repository artifacts. ([arXiv][8])

The memory mechanisms themselves are also getting more concrete. Improving Code Localization with Repository Memory shows that commit history and linked issues can be turned into non-parametric repo memory that improves localization on both SWE-bench Verified and SWE-bench Live. SWE-Exp stores structured experience from successful and failed repair attempts and reports a **73.0% Pass@1** result with Claude 4 Sonnet on SWE-Bench Verified, while MemGovern converts GitHub history into **135K governed experience cards** and improves Verified resolution by **4.65%**. These papers all point in the same direction: valuable project memory is not generic chat history; it is **structured repository experience** distilled into reusable forms. ([arXiv][9])

The benchmark side makes the need even clearer. SWE-STEPS shows that isolated pull-request evaluation can overstate performance by **up to 20 percentage points** because it ignores spillover from earlier poor choices. SlopCodeBench shows no tested agent solves any problem end-to-end and that verbosity rises in **89.8%** of trajectories while structural erosion rises in **80%**. SWE-CI similarly shifts evaluation from one-shot correctness to maintainability over time, with tasks spanning an average of **233 days** and **71 commits**. So the real open problem is no longer “can the agent remember previous turns?” It is “can the agent maintain a **clean, durable, repository-grounded project state** that resists drift, technical debt, and spec loss over long engineering trajectories?” ([arXiv][10])

## 4. Execution-grounded feature reasoning

This direction now looks essential, not optional. FeatureBench shows that feature-oriented software development is much harder than standard issue-resolution benchmarks suggest. It derives tasks automatically by tracing from unit tests along a dependency graph, builds **200 tasks** and **3,825 executable environments**, and finds that a model with **74.4%** resolved rate on SWE-bench achieves only **11.0%** on FeatureBench. That is a major warning that static repo understanding and feature-level executable understanding are not the same capability. ([arXiv][11])

ReCUBE and Gistify both support that conclusion from different directions. ReCUBE checks whether repository context can actually be transformed into code that works under usage-aware internal and cross-file tests. Gistify asks the model to produce a **single, minimal, self-contained** file that reproduces the behavior of a repository functionality from an entrypoint, which forces the model to follow execution flow and compress only the essential runtime-relevant components. This is an especially useful reframing because it turns “repo understanding” into a behavioral test rather than a static reading test. ([arXiv][5])

The infrastructure papers make this direction look scalable. RepoST shows that **sandbox testing** can provide execution feedback without rebuilding entire repositories, producing RepoST-Train with **7,415 functions from 824 repositories** and improving Pass@1 on both HumanEval and RepoEval. SWE-Next scales execution-grounded task construction even further by mining real merged PRs, keeping only commit pairs that show **strict test improvement without regressions**, and producing **2,308 self-verifying instances** from **3,971 seed repositories** and **102,582 candidate commit pairs**. These papers suggest that lightweight execution-grounded loops are becoming practical enough to be part of mainstream repo-agent design. ([arXiv][12])

What this adds to the original research subject is a stronger claim: a repo agent does not really “understand” a codebase unless it can track **feature behavior through callers, tests, and runtime paths**. Static graph retrieval and compressed summaries are necessary, but they are not sufficient. The open problems here are feature-boundary discovery, caller-trace selection, executable abstraction, lightweight environment construction, and how to summarize runtime evidence back into the retrieval and memory layers. ([arXiv][11])

## 5. Leakage-resistant evaluation

The deeper benchmark literature makes this look like a core research problem, not just an evaluation hygiene issue. SWE-QA-Pro explicitly targets **long-tail repositories with executable environments**, filters out questions solvable by direct-answer baselines, and finds that agentic workflows outperform direct answering by about **13 points** for Claude Sonnet 4.5. StackRepoQA reaches a similar conclusion from another angle: it is built from **1,318 real developer questions across 134 Java repositories** and shows that some apparent success comes from **verbatim answer reproduction** rather than true repository reasoning. Those two papers together make a strong case that benchmark design can easily overstate repo understanding. ([arXiv][13])

The contamination issue is broader than QA. The SWE-Bench Illusion paper shows that state-of-the-art models can identify buggy file paths from issue descriptions alone at up to **76%** on SWE-Bench Verified, but only **53%** on repositories outside SWE-Bench, which is strong evidence that memorization or dataset-specific artifacts can inflate apparent reasoning ability. Saving SWE-Bench pushes the same point from a product-realism angle: when formal benchmark prompts are mutated into more realistic user-style queries, some models’ performance drops by **more than 50%** relative to the standard public benchmark setup. ([arXiv][14])

The newest methodology papers add two more important controls. The time-consistent repository benchmark formalizes evaluation around a historical snapshot at time T0 and only uses repository knowledge available before T0, while tasks are derived from future pull requests; it also shows that **prompt construction is a first-order benchmark variable**, with file-level F1 changing monotonically across prompt granularities. RepoMod-Bench hides all tests from agents and uses **implementation-agnostic black-box testing**, then shows a severe scaling collapse from **91.3%** average pass rate below **10K LOC** to **15.3%** above **50K LOC**. These are strong examples of what more trustworthy repo-agent evaluation could look like. ([arXiv][15])

So leakage-resistant evaluation now looks like a design target with several concrete requirements: long-tail repositories, post-cutoff snapshots, prompt-control studies, hidden or mutation-resistant tests, executable verification, and settings that genuinely require exploration rather than recall. Without those controls, it is very hard to know whether gains in retrieval, memory, or execution reasoning are real. ([arXiv][13])

## What this now says about the original research subject

The original question was about getting **faster and more complete repo understanding with fewer tokens**. After this deeper research, the stronger formulation is:

**the key problem is not “how do we show the model more of the repo?”**
It is **how do we let the agent stop early with the right evidence, use that evidence faithfully, preserve project state over time, and verify its understanding against feature behavior under evaluation protocols that do not reward shortcuts.** Retrieval is still the first layer, but it now sits inside a larger architecture of **evidence sufficiency, evidence utilization, project-state continuity, execution-grounded reasoning, and leakage-resistant measurement.** ([arXiv][1])

If I compress the research agenda down to the most promising concrete questions, they would be these:

1. **Can we build a sufficiency estimator** that predicts when the current evidence set is enough to answer or edit safely, so the agent can stop searching early? ([arXiv][16])
2. **Can we build utilization-aware attribution** that scores which retrieved artifacts actually changed the answer probability or patch correctness? ([arXiv][3])
3. **Can project memory be made repository-native**, storing design commitments, commit-history abstractions, regression history, and evolving spec state instead of generic conversational summaries? ([arXiv][7])
4. **Can execution evidence be compressed into reusable signals** for retrieval and planning, rather than treated only as final pass/fail feedback? ([arXiv][12])
5. **Can evaluation be made simultaneously realistic, executable, and leakage-resistant** so that improvements in the first four areas are scientifically credible? ([arXiv][13])

The strongest overall conclusion is still simple:

**longer context is not the main answer.**
The more promising path is **better evidence selection, better evidence use, better project-state memory, better execution-grounded feedback, and better benchmarks.** ([arXiv][1])
